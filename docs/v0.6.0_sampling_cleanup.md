# v0.6.0 sampling cleanup â€” planned fixes

Working doc capturing three connected fixes that need to land on `az/model-params` before v0.6.0 is finalized. Decisions made in session 2026-04-27. Authoritative spec for the cleanup PR(s).

## Context

`az/model-params` introduces `forge.clients.sampling_defaults.MODEL_SAMPLING_DEFAULTS` and threads `top_p/top_k/min_p/repeat_penalty/presence_penalty` kwargs through `OllamaClient` and `LlamafileClient`. The eval surface (`tests/eval/batch_eval.py`, `tests/eval/eval_runner.py`) splats `get_sampling_defaults(model)` into client constructors and works correctly.

Audit on 2026-04-27 found three issues that are in-scope to fix before v0.6.0:

1. **Proxy doc/code drift.** Both `sampling_defaults.py:3-8` and `MODEL_GUIDE.md:130-145` claim the proxy forwards inbound sampling params. The code does not â€” it discards them.
2. **No first-class API for "use recommended params."** Consumers must `import get_sampling_defaults` and splat manually. There's no constructor-level affordance, and unknown models silently return `{}`.
3. **LlamafileClient and OllamaClient hardcode `temperature: float = 0.7`** in their constructors and unconditionally include it in every outbound body. "No sampling kwargs passed" does not mean "backend defaults" â€” it means `T=0.7` always wins.

Issue 3 is the load-bearing one: it's why every prior forge-code eval run sampled at `T=0.7` regardless of model card recommendation. Fixing it is necessary for the rig-03 sanity re-run to be a meaningful comparison.

## Issue A: Proxy is not a pass-through

### Current state

`src/forge/proxy/handler.py:75-99` extracts only `messages`, `tools`, `stream`, `model` from the request body. Inbound `temperature`, `top_p`, etc. are silently discarded. The proxy then calls `client.send(api_messages, tools=...)` â€” never passes sampling. Whatever the proxy's pre-built client was constructed with applies (and `proxy.py:166-188` constructs without sampling kwargs, so `LlamafileClient`'s hardcoded `T=0.7` applies â€” see Issue C).

### Fix (A1: thin pass-through)

Make the proxy honor inbound body sampling params. The proxy is meant to make local forge look like an OpenAI-compatible endpoint; OpenAI honors body sampling; forge proxy should too.

**Code changes:**

- `src/forge/proxy/handler.py`: in `handle_chat_completions`, extract a sampling dict from the body â€” `temperature`, `top_p`, `top_k`, `min_p`, `repeat_penalty`, `presence_penalty`, `seed` (and any other recognized OpenAI-compatible sampling fields). Plumb through to `client.send()` and `run_inference()`.
- `src/forge/clients/base.py` (`LLMClient.send` signature) and `src/forge/core/inference.py` (`run_inference` signature): add an optional `sampling: dict[str, Any] | None = None` kwarg. When provided, override the client's stored sampling for that call only â€” do NOT mutate `self`.
- `src/forge/clients/llamafile.py` and `ollama.py`: `send()` / `send_stream()` / native+prompt variants accept and apply per-call `sampling` overrides at body construction time. Call-scope, not instance-scope.
- `src/forge/clients/anthropic.py`: ignore the `sampling` kwarg (Anthropic SDK owns sampling internally; we don't currently expose those knobs).

**Proxy construction stays simple.** Sites at `proxy.py:166`, `:186`, `:188` continue to pass no sampling kwargs (and after Issue B, no `recommended_sampling` either). The proxy's client is a "blank slate" â€” body params are the only sampling source.

### Doc rewrites

- `src/forge/clients/sampling_defaults.py:3-8`: rewrite the proxy paragraph. New text: *"Proxy callers do not consult this map. The proxy plumbs through whatever sampling params the inbound request carries (OpenAI-compatible body fields). For per-model recommended sampling in proxy mode, the calling client looks up the map and includes the params in the request body."*
- `docs/MODEL_GUIDE.md:130-145`: same rewrite, expanded.
- `docs/USER_GUIDE.md`: add a short "proxy mode" subsection showing a request-body sampling example.

## Issue B: First-class opt-in for recommended sampling

### Decision

Constructor flag `recommended_sampling: bool = False` (opt-in, default off). Explicit. No magic.

| Caller passed | Model in map? | Behavior |
|---|---|---|
| `recommended_sampling=True` | yes | Splat map values; caller's explicit kwargs override field-by-field |
| `recommended_sampling=True` | no | **Raise** `UnsupportedModelError(model)` |
| `recommended_sampling=False` (default) | yes | One-shot **INFO** log: *"Recommended sampling params exist for `<model>`; pass `recommended_sampling=True` to use them."* No behavior change. |
| `recommended_sampling=False` (default) | no | Silent. Same as today. |

**Why opt-in, not opt-out.** Most forge users want backend-default sampling. Auto-application would silently change behavior for any consumer who upgrades to v0.6.0. Explicit opt-in matches user mental model and preserves Principle #1 â€” if you ask for the thing, forge either gives it to you or fails loud.

**Why raise on opt-in + unknown model.** When the caller declares "I want recommended sampling," falling through to backend defaults is a silent failure of intent. Raising forces a clear fix: either add the model to the map (with an HF-card URL comment) or drop the flag.

**Why an info log on opt-out + supported model.** Visibility without intrusion. Most libraries log deprecations/upgrades loudly; this is the same pattern. The log fires once per (model, process) pair.

### Code changes

- New exception `forge.errors.UnsupportedModelError` (check `errors.py` first for an existing fit before adding).
- `src/forge/clients/sampling_defaults.py`: split into two functions â€” getting and applying are separate concerns.
  - `get_sampling_defaults(model: str) -> dict` â€” pure lookup. Returns the map value (copy) or `{}` for unknown. **No logging, no raising.** This loses the current one-shot warning that lived inside this function; the warning logic moves into `apply_sampling_defaults`.
  - `apply_sampling_defaults(model: str, *, strict: bool) -> dict` â€” policy layer. Calls `get_sampling_defaults` internally and applies the four-quadrant behavior matrix:
    - `strict=True` + known model: return the dict
    - `strict=True` + unknown model: **raise `UnsupportedModelError`**
    - `strict=False` + known model: one-shot **INFO** log; return `{}`
    - `strict=False` + unknown model: return `{}` (silent)
- `src/forge/clients/llamafile.py` (`LlamafileClient.__init__`):
  - Add `recommended_sampling: bool = False` param.
  - Always call `defaults = apply_sampling_defaults(model, strict=recommended_sampling)` early in `__init__` â€” this drives both opt-in (returns map values or raises) and opt-out-with-known-model (fires the info log, returns `{}`).
  - Merge caller kwargs over defaults so explicit kwargs win field-by-field: `for k, v in defaults.items(): if locals_kwarg(k) is None: setattr(self, k, v)`. Or equivalent using a small helper â€” point is caller's non-None kwargs take precedence.
- `src/forge/clients/ollama.py`: same.
- `src/forge/clients/anthropic.py`: accept `recommended_sampling` for symmetry, but it's a no-op â€” log a debug message if `True` is passed (Anthropic doesn't expose sampling through forge today).

**Why split the functions.** Lookup and policy are different concerns. `get_sampling_defaults` becomes a clean primitive for any caller who wants to see what's in the map without triggering policy (debugging, doc generation, etc.). `apply_sampling_defaults` is the only thing client constructors call.

### Migration impact for known consumers

- `tests/eval/batch_eval.py` and `tests/eval/eval_runner.py`: replace `**get_sampling_defaults(config.model)` splats with `recommended_sampling=True`. Cleaner, same outcome â€” and now goes through `apply_sampling_defaults`, so unknown models in the eval list will raise instead of silently running with backend defaults.
- `forge.proxy.proxy`: no change at construction sites. The proxy intentionally stays "blank-slate" + body-pass-through (Issue A).
- `src/forge/server.py:478` (one-shot Ollama client for budget resolution): no change â€” sampling irrelevant.
- **External (out-of-tree) consumers** (forge-code, NORA, anything else): require explicit `recommended_sampling=True` to start using map values. Pre-v0.6.0 code that didn't splat will continue to behave identically (modulo the new info log on supported models â€” silenceable by passing `recommended_sampling=False` explicitly if the noise matters).

## Issue C: Drop hardcoded `temperature=0.7` from client constructors

### Current state

```python
# src/forge/clients/llamafile.py:131
temperature: float = 0.7,

# src/forge/clients/ollama.py:46
temperature: float = 0.7,
```

`LlamafileClient` send sites (`:251`, `:433`, `:499`) unconditionally include `"temperature": self.temperature` in the body. `OllamaClient._build_options` (`:75`) does the same. Result: a caller who passes only `model=...` gets `T=0.7` sent on every request, regardless of backend defaults or model recommendations.

### Fix

Drop the hardcoded default. Match the pattern already used for the other sampling fields (None = don't send).

```python
# Both clients:
temperature: float | None = None,
```

In `LlamafileClient`'s three send sites, move `"temperature"` out of the literal body dict and into `_apply_sampling`, conditionally added when non-None. In `OllamaClient._build_options`, same â€” initialize `opts: dict = {}`, conditionally add temperature.

### Behavior change

| Before | After |
|---|---|
| `LlamafileClient(model=m)` â†’ sends `temperature=0.7` | `LlamafileClient(model=m)` â†’ sends nothing for temperature â†’ backend default applies |
| `LlamafileClient(model=m, temperature=0.5)` â†’ sends 0.5 | unchanged |
| `LlamafileClient(model=m, recommended_sampling=True)` (Issue B) â†’ sends map value | unchanged (map provides temperature) |

**This is a real backwards-incompatibility window** for any consumer that constructed a client with no sampling kwargs and was depending on `T=0.7`. Note in CHANGELOG: callers who want `T=0.7` must now pass it explicitly.

- **Anthropic sampling.** AnthropicClient exposes no sampling kwargs today. Adding them is a separate feature, not a cleanup.

## Test plan

### New tests

`tests/unit/test_sampling_defaults.py`:
- `get_sampling_defaults(model)` for known model: returns map value (existing test, retained)
- `get_sampling_defaults(model)` for unknown model: returns `{}`, no log, no raise (retain shape; drop the one-shot warning assertion since the warning moves to `apply_sampling_defaults`)
- `apply_sampling_defaults(model, strict=True)` known model: returns map value
- `apply_sampling_defaults(model, strict=True)` unknown model: raises `UnsupportedModelError`
- `apply_sampling_defaults(model, strict=False)` known model: returns `{}`, fires one-shot INFO log (only once per (model, process))
- `apply_sampling_defaults(model, strict=False)` unknown model: returns `{}`, no log

`tests/unit/test_llamafile_client.py` (mirror in `test_ollama_client.py`):
- `LlamafileClient(model=m)` (no kwargs): no `temperature` field in outbound body â€” covers Issue C
- `LlamafileClient(model=m, temperature=0.5)`: `temperature=0.5` in body
- `LlamafileClient(model=m, recommended_sampling=True)` for known model: map values in body
- `LlamafileClient(model=m, recommended_sampling=True)` for unknown model: raises
- `LlamafileClient(model=m, recommended_sampling=True, temperature=0.99)`: caller's 0.99 wins over map
- `LlamafileClient(model=m, recommended_sampling=False)` for known model: info log fires once

`tests/unit/test_proxy_handler.py`:
- Inbound body with `temperature=0.5` is plumbed through to `client.send()` as `sampling={"temperature": 0.5, ...}`
- Inbound body with no sampling fields: `client.send()` receives `sampling=None` (or `{}`)
- Per-call sampling override does not mutate the client's instance state (run two requests, second one with no sampling, should not see first request's params)

### Existing tests that need updates

- Any client test that constructs `OllamaClient(model=m)` or `LlamafileClient(model=m)` and asserts a specific outbound `temperature` field â€” those will need updates per Issue C.
- `test_sampling_defaults.py`'s `test_unknown_model_logs_once` will need to be moved to test `apply_sampling_defaults(model, strict=False)` for a known-but-not-opted-in model (the new home of the one-shot log). The old "unknown model logs warning" test goes away â€” `get_sampling_defaults` no longer logs.

### Added
- Per-model recommended sampling defaults (`MODEL_SAMPLING_DEFAULTS`) sourced from
